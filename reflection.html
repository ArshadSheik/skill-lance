Critical Evaluation

AI Reflection Log

This page records how AI tools were used during planning, coding, and editing. The goal is not only to describe help received, but also to judge its reliability, limitations, and ethical implications in a university coding project.

How AI Was Used

I used AI primarily as a drafting and explanation assistant. It helped brainstorm page layouts, propose quiz logic, suggest wording for the tutorial, and review whether all assignment requirements had been addressed. I treated the output as a starting point rather than a finished solution.

For example, I asked for ideas on how to connect four pages into a single learning journey and how to implement a browser navigation warning on an unfinished quiz. The suggestions were useful for structure, but they still needed testing, simplification, and adaptation to my own coding style.

Example Prompts and Outputs

Prompt 1

"Design a four-page educational web app about HTML, CSS and JavaScript with a consistent theme."

The AI produced a site structure with a tutorial, quiz, reflection page, and CV page. This was useful because it mirrored the assignment brief and suggested a clearer navigation flow.

Prompt 2

"Show JavaScript that loads quiz questions from JSON, shuffles them, and scores the answers."

The output included fetch logic, randomisation, dynamic rendering, and answer checking. I used the general pattern but rewrote parts of it so the code matched my page structure and naming conventions.

Prompt 3

"What risks should I discuss in an AI reflection for a university web development task?"

The AI highlighted hallucinated code, hidden plagiarism, insecure snippets, and over-reliance. That helped me move beyond description and produce a more critical reflection.

Corrections and Changes

Several AI-generated suggestions were incomplete or slightly wrong. One early draft used hard-coded quiz questions inside JavaScript, which would have failed the requirement for AJAX-loaded local data. I corrected this by moving the question set into a separate JSON file and fetching it at runtime.

Another suggestion proposed using browser storage without checking support or error handling. I added defensive code to avoid crashes, made the history rendering clearer, and ensured the unload warning is removed after successful submission.

Limitations and Reliability

AI was strongest when explaining common web concepts and offering starter code patterns. It was weaker when handling assignment-specific details, because it sometimes ignored small but important constraints such as using a public API for rewards or including a clear references section on the CV page.

Because of this, every significant suggestion had to be cross-checked against the assignment brief and tested in the browser. The biggest limitation was confidence without certainty: the output often sounded correct even when it needed revision.

Ethical Considerations

Using AI in coursework creates both opportunities and risks. It can improve productivity and reduce frustration, but it may also tempt students to submit work they do not fully understand. That would undermine learning outcomes and make it difficult to explain or maintain the final code.

To use AI responsibly, I needed to remain the author of the final work, verify claims, rewrite unclear code, and acknowledge where AI contributed ideas or drafts. I also needed to avoid copying code blindly, especially where privacy, accessibility, or academic integrity could be affected.

Key Takeaways

Working with AI tools taught me several important lessons:

  • AI is a powerful starting point, not a final solution
  • Always verify generated code against requirements
  • Testing in real browsers reveals issues AI cannot predict
  • Understanding code is essential for debugging and maintenance
  • Transparent attribution builds trust and academic integrity
