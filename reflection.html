Critical Evaluation

AI Reflection Log

This page documents how AI (Claude by Anthropic) was used during the development of SkillLance. Each use case includes the actual prompt given, a summary of the AI output, changes I made, and a critical evaluation of the AI's reliability.

Use Case 1: Debugging the Quiz AJAX Loading

Prompt

"My quiz page shows 'Loading questions…' but the questions never appear. I'm using fetch('questions.json') to load them. The file is at data/questions.json relative to the project root. The HTML page is in the root. What's wrong?"

AI Output Summary

The AI correctly identified that the fetch path was wrong — it should be ./data/questions.json instead of just questions.json. It also suggested adding error handling with .catch() and checking the Network tab in DevTools to verify the 404 error.

Changes I Made

I adopted the path fix directly. I also added if (!response.ok) throw new Error() to catch HTTP errors more explicitly, and added a loading skeleton UI that the AI did not suggest.

Use Case 2: Creating CSS Animations for 3D Card Hover

Prompt

"Write CSS for a card component that has a 3D tilt effect on hover using CSS transforms. It should use perspective and rotateX/rotateY, feel smooth, and work without JavaScript."

AI Output Summary

The AI provided a CSS-only solution using transform-style: preserve-3d and perspective: 800px with hover transforms like rotateX(5deg) rotateY(-3deg). However, the CSS-only version could only produce a fixed tilt direction — it couldn't follow the mouse.

Changes I Made

I kept the CSS as a base hover effect but also added a JavaScript mousemove listener that calculates tilt based on cursor position within the card. The JS dynamically sets rotateX and rotateY based on the mouse offset from centre, producing a much more natural feel. On mouseleave, the card resets smoothly.

Use Case 3: Implementing the beforeunload Warning

Prompt

"How do I show a browser warning when the user tries to close or navigate away from a quiz page, but only after they've started answering? And how do I remove the warning after they submit?"

AI Output Summary

The AI explained the beforeunload event and showed a handler that sets e.returnValue = ''. It suggested using a boolean flag (quizStarted) set to true on the first radio button change, and clearing the listener with removeEventListener after form submission.

Changes I Made

The AI's initial code attached the listener immediately on page load. I changed this so addEventListener('beforeunload', ...) is only called after the "Start Quiz" button is clicked and the first answer is selected. This avoids false warnings when the user is just viewing the quiz page without starting.

Use Case 4: Particle Canvas Background Animation

Prompt

"Create a subtle particle background effect using HTML5 canvas. Small dots should float slowly and connect with faint lines when near each other. Keep it performant and not distracting."

AI Output Summary

The AI produced a working canvas animation with particles and connection lines. However, it used 120 particles with a 200px connection radius, which caused visible performance issues on smaller screens and made the background too busy.

Changes I Made

I reduced the particle count to scale with screen area (capped at 60), lowered the connection radius to 150px, reduced opacity values, and added a resize listener that recreates particles on window resize. I also made the particle color adapt to the current theme (dark vs light) by reading the data-theme attribute.

Use Case 5: Deepening Interactivity & Rebuilding the CV Page

Prompt

"Rebuild my CV page completely. Change it from a simple top-down list to a modern 2-column Grid layout with a sidebar. Add an 'Experience' section. Also, update my Quiz engine so that if the user fails, it triggers an AJAX fetch to a public Dad Joke API as a 'penalty'."

AI Output Summary

The AI provided a comprehensive overhaul plan. It generated HTML utilizing display: grid for a .cv-layout, styling the sticky sidebar and timeline elements with custom CSS. It also accurately wrote the fetch() block for the icanhazdadjoke API to trigger only when the score was below the 70% threshold.

Changes I Made & Critical Evaluation

While the layout generated by the AI was visually stunning, it overwrote some of my original color variables. I had to manually merge the AI's 2-column CSS into my existing style.css to ensure the custom Light/Dark mode toggle (which I had already perfected) wasn't broken by the new CSS classes. The AI's code for the Dad Joke API was flawless, correctly demonstrating how to pass the Accept: application/json header to receive JSON instead of HTML.

Identifying a Subtle AI Error

When I asked the AI to implement array shuffling for quiz randomisation, it initially suggested [...arr].sort(() => Math.random() - 0.5). This looks correct but produces a biased shuffle — some orderings are more likely than others because sort() comparison functions must be consistent, but Math.random() is not.

I detected this by researching shuffle algorithms on MDN and Stack Overflow after noticing certain questions seemed to cluster together. I replaced it with the Fisher-Yates algorithm, which swaps elements from the end of the array backwards, producing a truly uniform random permutation. The AI did not flag the bias in its own suggestion, which demonstrates how generated code can be syntactically correct but semantically flawed.

Limitations and Reliability

AI was strongest at explaining standard web APIs (fetch, localStorage, DOM methods) and generating boilerplate CSS. It was weakest when dealing with project-specific constraints — file paths, assignment requirements, and integration with existing code. Every AI suggestion required manual testing in the browser before I could trust it.

The biggest risk was confidence without accuracy: the AI's output always sounded authoritative, even when it contained errors (wrong paths, biased shuffles, over-heavy animations). Without critical evaluation, these errors would have been submitted as-is.

Ethical Considerations

Using AI in coursework requires maintaining genuine authorship. I ensured this by treating AI output as draft material — always testing, modifying, and sometimes rejecting suggestions. The final code reflects my understanding because I can explain every function, debug issues independently, and adapt the code to new requirements.

Transparency matters: all AI usage is documented here and in the references section of the CV page. Students who submit AI output without understanding it risk failing the demo session, where they must explain their code. More fundamentally, uncritical reliance undermines the learning outcomes the assessment is designed to measure.

Key Takeaways

  • AI is most useful for explaining APIs and generating starter patterns, not for project-specific integration
  • Always test AI-generated code in a real browser — file paths, API responses, and edge cases often fail
  • Seemingly correct algorithms (like sort-based shuffle) can have subtle mathematical biases
  • Performance tuning (particle counts, animation intensity) requires human judgement, not just code generation
  • Understanding the code is essential — for debugging, for the demo, and for genuine learning
